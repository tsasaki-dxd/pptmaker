# 02. UI / UX 設計書

**文書名**：SlideForge 詳細設計 — UI / UX
**版**：Draft v0.1
**参照**：`SlideForge_概要設計書.md` §2 / §6

---

## 1. 目的とスコープ

Web UI（Next.js / React）の画面・遷移・インタラクションを設計する。プロンプト慣れしていないユーザーでも定型提案書を量産できる体験を目指す。

---

## 2. UX 設計原則

| 原則 | 意図 |
|---|---|
| 段階的開示 | 一度に見せる選択肢は最小限。必要に応じて深く掘れる |
| 差分可視化 | LLM の変更は必ず「前後比較」で提示し、受諾/却下可能に |
| 巻き戻し容易 | 任意時点に戻せる。間違いを恐れずに試行できる |
| 既定値優先 | 迷いやすい選択（色、レイアウト、図表タイプ）に安全な既定を提供 |
| 透明な AI | どのスライドが LLM で生成・修正されたか可視化、編集権はユーザーに |

---

## 3. 主要ペルソナ

| ペルソナ | 役割 | 主な行動 | 主な課題 |
|---|---|---|---|
| 営業担当 佐藤 | 受注前の提案書作成 | 週3件の提案書を量産 | 時間不足、品質ばらつき |
| コンサル 鈴木 | 顧客向け報告書作成 | 月10件の報告書、深い構成 | 構成の検討に時間がかかる |
| テンプレ管理者 田中 | 社内テンプレ整備 | 四半期に1回の更新 | 個人のPPT編集スキルに依存 |
| 閲覧者 山田 | マネージャ | レビュー、フィードバック | 細かい修正の指示が大変 |

## 8. 修正指示 UI の設計判断

| 方式 | 長所 | 短所 |
|---|---|---|
| A. チャット主 | 文脈保持しやすい、複数スライドに跨がる修正 | 対象指定が曖昧になりがち |
| B. スライド上選択＋指示 | 対象明確、直感的 | 跨がる修正が難しい |
| C. 併用 | 両方の利点 | 実装コスト高 |

**結論：C（併用）。主はチャット、補助としてプレビュー上の要素クリックで「この表を対象に」等のコンテキスト付与**。

- プレビュー上で要素クリック → チャット欄に `@slide4.table` 等のコンテキストチップが自動挿入
- チャットのみでも完結できる設計

## 9. 差分プレビューの UX

- **並列表示（Before / After）を標準**、必要に応じて重ね表示（Onion Skin）へ切替
- 変更箇所を色付き矩形でハイライト（追加=緑、削除=赤、変更=青）
- 「この修正を適用 / 却下」ボタンで 1 クリック操作
- 却下した修正はチャットに「却下済み」として残し、再修正の文脈に使える

## 10. エラー・空状態・ローディング

| 場面 | 体験 |
|---|---|
| LLM 生成中（〜30秒） | 骨格ごとにストリーミングで段階表示。打ち切り可能 |
| LLM 失敗 | 原因カテゴリ（ネットワーク/スキーマ違反/コンテンツ拒否）と再試行導線 |
| レンダリング失敗 | 失敗スライドのみ XML 詳細をトグルで開示（サポート依頼ボタン） |
| 空状態（初回） | サンプルプロジェクトへの誘導、3分チュートリアル |
| アップロードサイズ超過 | 事前検知でエラー、圧縮提案 |

## 11. アクセシビリティ（WCAG 2.1 AA）

- キーボード操作：全フォーム Tab 可達、ショートカット（`⌘/Ctrl+K` で Command Palette）
- スクリーンリーダー：全画像に代替テキスト、チャート要素に ARIA 属性
- コントラスト比 4.5:1 以上（UI コンポーネント）。プレビュー内容はユーザーのテンプレ依存
- カラーオンリーで情報伝達しない（アイコン＋テキスト併記）
- フォーカスリングを視認可能な太さで
- 動きに敏感なユーザー向けに `prefers-reduced-motion` 対応
