## 2. 事業仮説

### 2.1 バリュープロポジション

- 提案書作成時間の **短縮**（現状 6 時間 → 2 時間目標）
- コーポレートテンプレート準拠の **品質均質化**
- 属人性の排除・ナレッジの蓄積（テンプレ／図表パターン）
- 日本語・日本のビジネス文脈への **最適化**

| Phase 2（顧客案件） | 5 社 × 50 名 | 800 件 |
| Phase 3（SaaS） | 50 社 × 30 名 | 6,000 件 |

## 4. 月次ランニングコスト試算

為替 1 USD = 150 円、2026 Q2 の AWS / Claude API 単価目安。ap-northeast-1。

### 4.1 設計方針

Phase 1 は **サーバーレス最小構成**（Lambda 中心、単一アカウント、単一 AZ、NAT/CloudFront/WAF 不採用）で立ち上げ、Phase 2 開始時点で再設計する（詳細は `SlideForge_概要設計書.md` §3）。この結果、Phase 1 の月次運用コストが過剰スケール設計時の 1/10 程度に圧縮される。

### 4.2 Phase 1 明細（50 名 / 月 200 提案書、単一アカウント・単一ステージ）

Phase 1 は **main マージ＝本番デプロイの単一ステージ**。Stg / Dev 環境は持たない。

| サービス | 構成 | 計算根拠 | USD | 円 |
|---|---|---|---:|---:|
| Lambda（API） | 3,000 invoc × 512MB × 2s | GB-s 従量 | $1 | 150 |
| Lambda Container（Render） | 200 × 3GB × 60s | LibreOffice 同梱 | $1 | 150 |
| Fargate Spot（バースト予備） | 緊急時のみ | — | $3 | 450 |
| API Gateway | 50k req × $1/M | — | $1 | 150 |
| RDS db.t4g.small Single-AZ | $0.027/h × 720 + 20GB | メタデータ | $20 | 3,000 |
| S3 | 3GB + リクエスト | — | $1 | 150 |
| SQS | 200 msg | 無料枠内 | $0 | 0 |
| Cognito | 50 MAU | 無料枠内 | $0 | 0 |
| CloudWatch | Logs 5GB + Alarm 10 | — | $4 | 600 |
| Secrets Manager | 3 秘匿 | $0.40 / secret | $2 | 300 |
| KMS | 2 CMK | $1 / CMK | $2 | 300 |
| CodePipeline V2 | 1 pipeline | — | $1 | 150 |
| CodeBuild | 月 200 分 | $0.005/min | $2 | 300 |
| Claude API | §4.3 参照 | Sonnet + Cache | $80 | 12,000 |
| **合計（Prod のみ）** | | | **$118** | **17,700 / 月** |

**Phase 1 で不採用**（Phase 2 以降で復活）：Stg 環境、NAT Gateway、CloudFront、ElastiCache、WAF、GuardDuty、Aurora Serverless v2、クロスリージョン DR、マルチアカウント。

バースト・為替バッファ込みで **月 2.5 万円**を予算化。

### 4.3 Claude API コスト試算

想定（`01_prompt_engineering.md` §8 より、1 提案書あたり）:
- input 17,500 tokens / output 6,500 tokens
- Prompt Caching 70% ヒット想定 → 有効 input ≒ 5,250 + 12,250×0.1 = 6,475
- Sonnet 4.6 想定単価：input $3 / output $15 / MTok（仮）
- 1 提案書：($3 × 6,475 + $15 × 6,500) / 1M ≒ $0.117
- 月 200 提案書：約 $23
- 修正 3 回平均・Eval・Haiku 併用で実測倍率 3〜4 倍想定 → **月 $80（約 12,000 円）**

### 4.4 Phase 2 / Phase 3 概算

Phase 2 開始時点でアーキテクチャを再設計（Aurora Serverless v2、ECS Fargate 常駐、WAF、CloudFront、マルチアカウント）するため、コストは非線形に増える。

| Phase | 月間生成 | 推定月額 | 主な増加要因 |
|---|---:|---:|---|
| Phase 1 | 200 件 | 約 1.8 万円 | Lambda サーバーレス最小・単一ステージ |
| Phase 2（5 社 × 50 名） | 800 件 | 40〜55 万円 | Aurora v2、ECS 常駐、WAF、Stg 環境復活、マルチアカウント |
| Phase 3（50 社 × 30 名） | 6,000 件 | 180〜250 万円 | CloudFront、GuardDuty、DR、ElastiCache |

### 4.5 Unit Economics

| 指標 | Phase 1 | Phase 2 | Phase 3 |
|---|---:|---:|---:|
| 1 提案書あたり原価 | 約 100 円 | 約 600 円 | 約 380 円 |
| 目標顧客単価（SaaS Standard） | — | — | 30,000 円 / 月 |
| 粗利率目標 | — | — | 60% |

## 7. ROI 分析（Phase 1）

### 7.1 現状

- 1 提案書平均作成時間：6 時間
- 営業担当の時間単価：7,500 円 / 時（600 万 / 年 × 係数 1.5 / 1,200 時）
- 月 200 件 × 6 時間 × 7,500 円 = **月 900 万円 の労務投下**

### 7.2 導入後

- 1 提案書平均 2 時間（目標）
- 月 200 件 × 2 時間 × 7,500 円 = 300 万円
- **時間削減価値：月 600 万円**

### 7.3 回収

- 月次ランニング 約 1.8 万円 vs 削減 600 万円
- 開発費 Phase 0〜1 合計 1,182 万円
- 回収期間 ≒ 2 ヶ月（稼働開始後、ランニング分のみなら初週で回収）

当然前提条件の精度は要検証。先行 10 件でタイムトラッキングを実施して更新する。

## 8. 競合分析

| サービス | 強み | 弱み |
|---|---|---|
| Microsoft 365 Copilot | PowerPoint 統合・既存資産活用 | 日本語品質・コーポレートテンプレ準拠の細かさが弱い、カスタム図表は限定 |
| Tome | 演出的プレゼン | 堅めのビジネス提案書には不向き |
| Gamma | 速い生成、編集しやすい | ブランドガバナンス弱、.pptx 互換に課題 |
| Beautiful.ai | 洗練テンプレ | 英語中心、テンプレ持ち込み制約 |
| SlideSpeak | ファイル中心の変換 | UI がやや簡素 |

**SlideForge の差別化**

1. コーポレートテンプレート**持ち込み準拠**（正規化＋プロファイル化）
2. 日本語特化・日本の提案書文化（セクション扉、会社概要、免責事項、金額表）
3. 図表プラグイン IF による**業務特化拡張**
4. 自社の社内業務での実績を顧客に提示できる

## 9. リスクとサニティチェック（ビジネス観点）

| # | リスク | 影響 | Mitigation |
|---|---|---|---|
| B1 | Copilot が無償バンドル化 | SaaS 化の価値毀損 | コーポレート持込・拡張性・日本語で差別化、大企業向けオンプレ路線を担保 |
| B2 | Claude API 価格上昇 / Rate Limit | 原価悪化 | モデル切替（Haiku 化）、Prompt Caching 強化、複数ベンダー対応余地を設計 |
| B3 | 社内利用が定着しない | Phase 1 失敗 | 先行ユーザー 5 名にデザインパートナー制、月次利用レビュー |
| B4 | 顧客テンプレが想定外パターン | 対応不可 | テンプレ登録段階で非対応を早期検知、段階的対応 |
| B5 | 情報漏洩事故 | 信用毀損 | セキュリティ設計書 §14 体制、Phase 3 前に外部監査 |
| B6 | 人員確保難 | 開発遅延 | Phase 2 以降で外部パートナー・オフショア活用案 |

## 10. ロードマップ（肉付け）

### 10.1 Phase 0（PoC 整理、2 週間）

- W1: 既存スクリプトのモジュール分割、テンプレプロファイルのスキーマ確定、Eval 初期セット 30 件
- W2: CI セットアップ（lint / unit / VRT の骨格）、Phase 1 要件確定、キックオフ準備

### 10.2 Phase 1 MVP バックログ（MoSCoW）

| 優先度 | 項目 |
|---|---|
| Must | Cognito 認証、テンプレ登録＋プロファイル、骨格生成（自由記述）、プレビュー、.pptx エクスポート、ロギング基盤 |
| Must | PowerPoint 互換 CI（Lint）、監査ログ、テナント分離 |
| Should | 構造化フォーム入力、修正指示（テキスト修正まで）、履歴閲覧 |
| Could | 既存文書取込、色調調整、差分プレビュー、チャット |
| Won't | 共同編集、モバイル、音声入力 |

### 10.3 Go/No-Go ゲート

| 判定 | ゲート基準 |
|---|---|
| Phase 1 → 2 | WAU ≥ 30 / 50、1 提案書所要時間 ≤ 3 時間、NPS ≥ 20、重大セキュリティ事故ゼロ |
| Phase 2 → 3 | 顧客案件 2 件以上導入、SLO 達成、ペネトレーションテスト合格、損益分岐の見通し |
| Phase 3 継続 | MRR 増加、解約率 ≤ 5% / 月、SOC 2 Readiness 合格 |

### 10.3.1 Phase 2 開始時のアーキテクチャ再設計

Phase 1 の最小構成は意図した技術的負債。Phase 2 キックオフで以下を実施する。

| 項目 | 移行内容 |
|---|---|
| DB | RDS t4g.small Single-AZ → Aurora Serverless v2（Multi-AZ） |
| API 実行基盤 | Lambda → ECS Fargate 常駐（コールドスタート・15 分制限回避） |
| キャッシュ | なし → ElastiCache Redis |
| CDN | なし → CloudFront |
| WAF | なし → AWS WAF（OWASP Top10 マネージドルール） |
| 環境分離 | 単一ステージ（Prod のみ） → Dev/Stg/Prod の独立 AWS アカウント（Stg 環境を新設） |
| 監査・セキュリティ | 基本 → GuardDuty、Config、Security Hub |
| 移行工数 | Phase 2 の 4 週間のうち 1.5 週相当を割り当て |
